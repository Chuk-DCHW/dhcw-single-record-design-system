Elevation & Shadow

Three functional tokens — each with a named role. Shadow is reserved for elements that genuinely float above the page. Borders and surface colours handle structural hierarchy everywhere else.

Context

Why three tokens

GDS and NHS England use shadow in one place only. SR extends this deliberately and minimally.

GDS and NHS England are intentionally flat. GDS uses box-shadow: 0 2px 0 on the button component only — a press affordance, not a depth metaphor. NHS England uses the same pattern with a navy bottom shadow. Neither system defines cards, modal, or overlay shadows. This is deliberate: public service interfaces are built for potentially low-quality displays where subtle shadows are unreliable.

SR spans Blazor and .NET MAUI. Mobile UIs have established conventions around cards, bottom sheets, and dialogs that users expect to carry depth. Without defined shadow tokens, developers will invent values inconsistently. Three role-named tokens cover all current use cases and leave no ambiguity.

Shadow colour is navy-900, not black. rgba(27,41,74,…) reads as a tinted shadow on white surfaces — warmer and more on-brand than generic grey-black, and less harsh on clinical displays.

Rules

Usage Rules

  • Use sr.elevation.raised only when a border alone is insufficient to separate a surface — for example, a card floating on a similarly-coloured background.
  • Use sr.elevation.overlay for all floating layers: modals, side drawers, dropdown menus, and tooltips. Pair with a scrim where content behind should be obscured.
  • Do not stack multiple elevation tokens on a single element. An element has one elevation level or none.
  • Do not use shadow decoratively. If the shadow's purpose cannot be explained in one sentence, remove it and use a border instead.
  • Do not rely on shadow alone to indicate interactivity — pair with a border or sr.color.surface.* change.
  • Focus rings are managed by sr.color.border.focus, not by elevation. Do not use box-shadow for focus rings.
  • If a new use case genuinely requires a different shadow value, add a named token via DDR — do not use an ad-hoc value.
